## 0. 설계 원칙 (전체를 관통하는 것)

1. **못 믿을 신호는 수치로 쓰지 않는다.** 단안 웹캠의 z(깊이)는 추정치, 옆모습의 손 라벨은 원리적으로 불안정, 프레임 밖 팔꿈치의 Pose 추정은 소음 — 셋 다 계산에서 배제하거나 용도를 제한했다.
2. **문제를 알고리즘으로 억지로 풀기보다, 문제 자체를 쉬운 문제로 바꾼다.** 3D 왜곡은 촬영 자세 설계로 2D 문제로 변환했다.
3. **애매하면 추론이 아니라 실측으로 확정한다.** 거울상 보정 여부는 양손 콘솔 실측으로 결정했다(§6).
4. **같은 인식 엔진, 다른 태도.** 정밀 측정(체크)은 깐깐하게, 운동 가이드(루틴)는 관대하게 — 목적이 다르면 판정 철학도 달라야 한다.

## 1. 문제 정의: 웹캠으로 3D 관절을 잰다는 것

손목은 3차원으로 움직이는 관절이고 웹캠은 2D 센서다. 카메라 축 방향 움직임은 화면에서 소실된다(원근 단축). MediaPipe Hands가 z값을 제공하지만 단안 추정치라 노이즈·계통 오차가 커서 각도 계산에 쓸 수 없다.

**해법의 뼈대: 측정하려는 움직임이 카메라 평면 안에서 일어나도록 동작별 촬영 자세를 배정한다.**

| 동작 | 촬영 자세 | 이유 |
|---|---|---|
| 굽힘·폄 (flexion/extension) | **옆모습** (손날이 카메라) | 회전이 화면 평면에 그대로 투영 — 왜곡 없음 |
| 좌우 편위 (radial/ulnar) | 정면 (손바닥/손등이 카메라) | 편위는 정면에서 평면 내 회전 |
| 손가락 동작 (주먹·펴기·핀치 등) | 정면 | 거리 비율 기반이라 자세 관대 |

정면 자세에서 굽힘·폄을 재는 방법(길이 비율 역산, acos(L/L₀))도 설계 완료 상태로 `측정모듈_설계서_v2.md`에 보관되어 있다. **폐기가 아니라 보험**: 옆모습의 유일한 구조적 약점(자기가림으로 인한 인식 끊김)이 실사용 테스트에서 실제 문제로 확인될 경우 투입하는 플랜 B다.

## 2. 각도 계산의 기본 (공통)

- 손목점(랜드마크 0)→중지 MCP(랜드마크 9)를 잇는 선을 "시계 바늘"로 삼고, 화면상 방향각을 atan2로 계산한다. 이 구간은 손등의 뼈 구간이라 손가락 굽힘에 오염되지 않는다 (TIP을 쓰지 않는 이유).
- 절대 방향은 사람·자세마다 다르므로, **중립 캘리브레이션**으로 개인 0점을 찍고 이후 상대각(rel = 현재각 − 중립각)만 사용한다.
- 노이즈는 중앙값 버퍼 + 지수평활(EMA)로 억제한다.
- z 좌표는 어느 모드에서도 수치로 사용하지 않는다. (백업 설계인 정면 방식에서만 굴곡/신전 "부호" 판별에 z의 부호를 제한적으로 사용 — 신호의 신뢰 수준에 맞는 용도만 부여)

## 3. 모드 A — 손목 체크 (정밀 측정, 주 1회 권장)

목적: **정확한 숫자를 기록에 남긴다.** 절차의 엄격함으로 하드웨어 부재를 보상한다.

| 단계 | 방식 | 근거 |
|---|---|---|
| 손 결정 | **수동 선택** (왼/오 버튼, 마지막 선택 기억) | 옆모습 손은 좌우가 거의 동일하게 보여 MediaPipe 라벨이 원리적으로 불안정 → 확실히 아는 사용자에게 묻는다. 측정 중 전환 칩 제공, 전환 시 기캡처된 굽힘/폄 값 자동 스왑 |
| 거울상 보정 | 선택된 손이 오른손이면 rel 부호 반전 | 양손은 거울상이라 같은 굽힘도 화면 회전 방향이 반대. 해부학적 방향(굴곡/신전)이 기록에 필요하므로 보정 필수 |
| 중립 캘리브레이션 | 곧게 편 채 약 2초, **손이 보이는 동안만 진행** | 오염된 기준 거부 |
| 좌표 기준 | **usePose 켬**: Pose 모델의 팔꿈치→손목 벡터 대비 상대각 | 체크 화면은 팔꿈치가 보이는 거리에서 사용 → 전완 기준이 가능하고, 이것이 임상 goniometry와 같은 정의 |
| 보상동작 | Pose로 팔꿈치·어깨 이동 감지 시 해당 프레임 **무효 처리** | 어깨가 만든 가짜 각도가 기록을 오염시키면 안 됨 |
| 캡처 | 끝범위에서 일정 시간(약 2.2초) 유지 + 흔들림(7°↑) 시 리셋 + 유지 구간 중앙값 | 순간 노이즈가 "오늘의 기록"이 되는 것 방지 |
| 저장 | `{ v: 1, at, hand, flex, ext, rom }` | 손 구분(재활은 편측 문제 — 좌우 혼합 금지) + 스키마 버전(향후 마이그레이션 대비) |

## 4. 모드 B — 오늘의 루틴 가이드 (관대한 회수 세기, 매일)

목적: **운동을 실제로 하게 만든다.** 정확한 각도가 아니라 "움직였는가"를 본다. 명세서 원칙: "게임이 아니라 가이드."

| 요소 | 방식 | 근거 |
|---|---|---|
| 판정 | 왕복 감지: rel ≤ −24°(굽힘 래치) 후 rel ≥ +12°(복귀)면 1회 | 정확한 각도 무관심. **−24/+12 비대칭은 의도된 설계**: 옆모습에서 폄이 카메라에 잘 안 잡혀 폄 문턱을 일부러 낮춤 — 변경 금지 |
| 좌표 기준 | **usePose 끔**: 화면 절대각 단일 기준 | 가이드는 근접 사용이라 팔꿈치가 프레임 밖 → Pose 추정이 프레임마다 널뛰며 기준 좌표계가 뒤바뀜(§6 사건 1). 없는 신호를 붙잡지 않는다 |
| 거울상 보정 | **없음 (실측으로 불필요 확정)** | 가이드는 해부학 방향이 아니라 화면 방향의 왕복을 봄. 어느 손이든 시범을 따라 같은 방향으로 움직이므로 양손 부호 동일(§6 사건 3). 보정을 넣었다면 오히려 한쪽 손이 고장 |
| 중립 | 손이 보이는 프레임 14개 수집 시 확정. 8초 타임아웃에도 손 프레임 3개 미만이면 확정하지 않고 대기(건너뛰기는 상시 가능) | 시간만 지나면 커밋하던 옛 방식은 오염 기준 위험 |
| 보상동작 | **감지하되 무효 처리 안 함.** 지표는 Pose가 아니라 "손목점의 화면 이동량 ÷ 손 크기(손목~중지 MCP 거리)" — 비율 0.5 초과 시 comp 판정, 세션 비율만 집계(`lastCompRatio`) | ① 관대한 판정 철학: 자세가 완벽하지 않아도 수행은 인정(무효 선언이 아니라 큐잉 — 임상의 실제 행동) ② 손목만 굽히면 회전축인 손목점은 제자리라는 기하학적 사실만 사용 ③ 손 크기 정규화로 카메라 거리 무관 ④ 집계값은 향후 코칭 힌트("팔꿈치를 책상에 붙이면 더 좋아요") 재료 |
| 가시화 | 손 감지 칩(연속 5프레임 감지→초록, 12프레임 소실→회색), 관절점 오버레이(거울 반전 x=1−x + cover 크롭 보정), 회수 인정 시 진행 점 팝 + 테두리 반짝(0.55초) | 웹캠 앱 최대 불안 "되고 있는 거 맞아?" 해소. 추적 상실은 정직하게 표시 |
| 탈출구 | 15초 인식 실패 시 "손동작 없이 진행" 제공 | 인식 실패로 사용자를 가두지 않는다 |

트레이드오프 (수용함): usePose를 끄면 팔 전체 기울임이 rel에 섞인다 → 회수 크기가 과장될 수 있으나 왕복 감지에는 지장 없음. 관대한 판정 철학상 허용. 이 상황은 comp 비율이 포착하며 코칭 힌트의 대상이다.

## 6. 해결된 사건 기록 (근거 보존용)

### 사건 1 — rel −130° 유령 값
- 증상: 가이드에서 생리학적으로 불가능한 각도(−86~−132°) 관측
- 원인: `usePose`가 매 프레임 재평가되는 구조에서, 근접 사용 시 팔꿈치가 프레임 밖 → Pose 추정이 "있다/없다"를 오락가락 → 중립은 한 좌표계에서 찍혔는데 이후 프레임이 반대 좌표계로 흘러 두 좌표계의 차이(수십~90°)가 rel에 통째로 실림
- 해결: 가이드 경로에서 usePose 차단 (main.js 2곳: `tracker.update(..., {usePose:false})`, `startLoop(..., {pose:false})`). measurement.js 무수정

### 사건 2 — 전 프레임 comp=true (회수 0의 주범)
- 증상: 보상동작 감지가 상시 발동, 모든 프레임 무효 → 아무리 운동해도 회수 불가
- 원인: comp도 프레임 밖 팔꿈치의 Pose 추정에 의존 — 튀는 추정값이 상시 임계 초과 + 정규화 분모(팔꿈치~손목 거리)도 불안정
- 해결: 지표를 손목점 이동/손 크기 비율로 교체 + 가이드에서 무효 처리 제거(감지·집계는 유지)

### 사건 3 — 거울상 보정, 실측으로 불필요 판명
- 추정: 체크에서 필요했던 보정이 가이드에 없으니 반대 손은 회수 불가일 것
- 실측 (DEBUG_GUIDE 로그): 왼손 굽힘 rel≈−80/폄 +44 (label=Left 안정) · 오른손 굽힘 rel≈−141/폄 +88 (label=Right 안정) — **양손 부호 방향 동일, 양손 모두 회수 정상 인정**
- 해석: 가이드는 화면 방향 왕복을 재고, 사용자는 어느 손이든 시범과 같은 방향으로 움직이므로 거울상이 무효화됨. 오른손의 큰 크기는 팔 동반 이동(move 0.5~1.29)이 절대각에 섞인 것 — §4 트레이드오프 범위
- 결정: 보정 미구현. 근거를 커밋 메시지에 기록해 향후 "누락 버그"로 오인·재도입되는 것 방지
- 교훈: 추론만으로 보정을 넣었다면 한쪽 손을 망가뜨리는 진짜 버그가 됐다 — **실측이 이론을 이긴다**

## 7. 디버깅·확장 포인트

- **진단 로그**: `js/config.js`의 `DEBUG_GUIDE = false`. true로 바꾸면 `[guide-diag]`(label·rel·move·세션 comp 비율)와 `[flexExt]`(임계 통과·래치 상태) 로그 활성화. 지인 테스트에서 인식 문제 리포트 시 첫 번째 도구
- **comp 비율 → 코칭 힌트** (예정): `lastCompRatio`가 높은 세션에 "팔꿈치를 책상에 붙이면 손목에 더 잘 들어요" 표시 — 무효 처리 대신 큐잉이라는 §4 철학의 완성
- **정면 측정 모드** (보험): 옆모습 자기가림이 실사용 문제로 확인되면 `측정모듈_설계서_v2.md`의 방법 B(길이 비율 역산 + 3중 캘리브레이션 방어) 투입
- **고니오미터 검증** (예정): 실측 각도(15/30/45/60°)와 앱 측정값 대조 — 계통 오차 발견 시 보정 계수 도입. 임상 도구로 직접 검증 가능한 것이 이 프로젝트의 차별점
- **Pose 확장** (v2 후보): 체크 화면의 보상동작 감시 강화(어깨 들썩임 등) — Pose는 "팔꿈치가 보이는 화면"이라는 조건 하에서만

## 8. 한 문장 요약

**같은 눈(MediaPipe), 같은 바늘(atan2 상대각), 다른 태도** — 체크는 정확한 숫자를 위해 깐깐하게, 가이드는 운동을 시키기 위해 관대하게. 3D 한계는 자세 설계로 2D화하고, 오차는 절차로 누르고, 잔여 한계는 추세 중심 포지셔닝으로 흡수하며, 애매한 것은 실측으로 확정한다.
